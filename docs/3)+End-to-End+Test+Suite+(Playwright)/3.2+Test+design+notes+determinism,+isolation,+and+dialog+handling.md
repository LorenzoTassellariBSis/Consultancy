## 3) End-to-End Test Suite (Playwright)

### 3.2 Test Design Notes: Determinism, Isolation, and Dialog Handling

This section explains how the Playwright tests are structured to run reliably and independently, how timing issues and animations are handled, and how JavaScript dialog alerts are validated within the suite.

#### Determinism and Isolation

- Every test begins from a clean slate by navigating to the application’s entry point in a `beforeEach` hook.
- The call to `page.waitForLoadState('networkidle')` ensures that all network activity and dynamic content loading have settled before any assertions or interactions occur .

```javascript
test.beforeEach(async ({ page }) => {
  // Ensure a fresh page load for each test
  await page.goto('http://localhost:8080/index.html');
  // Wait until network activity subsides for determinism
  await page.waitForLoadState('networkidle');
});
```

#### Timeouts for UI Updates and Animations

- Many interactions (saving input, animating success classes, updating totals) trigger CSS animations or DOM changes that require a brief pause before verification.
- Tests use explicit time-based waits—commonly `await page.waitForTimeout(300)` or `600` milliseconds—to give the UI time to complete transitions .

```javascript
// After clicking Save, wait for the success animation and DOM update
await page.waitForTimeout(600);
await expect(page.locator('#week1-display')).toHaveText('5');
```

#### Dialog Handling

- Input validation in the app triggers a standard `alert` dialog when a value outside 0–7 is entered.
- Tests attach an explicit `page.on('dialog', …)` listener to intercept, validate, and accept the dialog, preventing unhandled promise errors and allowing the test to continue .

```javascript
page.on('dialog', async dialog => {
  // Confirm the alert message and dismiss it
  expect(dialog.message()).toContain('Please enter a value between 0 and 7 days');
  await dialog.accept();
});

// Trigger validation
await page.locator('#week1-input').fill('8');
await page.locator('.week-card').first().locator('button').click();
```

#### Known Brittleness and Recommended Stability Improvements

- Hardcoded `waitForTimeout` durations can lead to flakiness in slower environments or if animations are tweaked.
- The suite’s README advises adopting `data-testid` attributes for selectors and replacing arbitrary waits with targeted state checks (e.g., `await expect(locator).toHaveClass('…')` or `await locator.waitFor()`) to increase resilience .
- Transitioning to explicit waiting for specific element states or network responses will further stabilize tests without relying on fixed delays.
